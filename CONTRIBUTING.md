## How to Contribute

### 1. Branch Workflow
- Fork the repository to your personal GitHub account.
- Clone your fork locally.
- Create a new branch for your feature or bug fix: `git checkout -b feature/your-feature-name` or `git checkout -b fix/your-bug-fix`.
- Do not make changes directly on the `master` or `main` branch of your fork.

### 2. Coding Style
- Follow the standard Dart and Flutter formatting guidelines.
- Use `flutter format` before committing.
- Ensure your code is free of linting errors by running `flutter analyze`.
- Keep the UI consistent with the Material 3 design system and existing SafeKey aesthetics.

### 3. Commit Conventions
- Write clear, descriptive commit messages.
- Start your commit message with a capitalized verb (e.g., `Add`, `Fix`, `Update`, `Refactor`).
- Keep the first line concise (under 50 characters). Provide more details in the commit body if necessary.

### 4. Pull Request Requirements
- Push your branch to your fork and submit a Pull Request (PR) against the main repository.
- Provide a clear and detailed description in your PR outlining what changes were made and why.
- Include screenshots or screen recordings if your PR involves UI changes.
- Ensure all CI/CD checks (if applicable) pass.
- Be open to feedback and ready to make requested adjustments during the review process.

### 5. Issue Reporting
- Before opening a new issue, please search existing issues to avoid duplicates.
- Provide clear steps to reproduce the bug.
- Include information about your device, OS version, and SafeKey version.
- Use issue templates if provided.

### 6. Contributor Expectations
- Treat everyone with respect (see our [Code of Conduct](CODE_OF_CONDUCT.md)).
- Do not submit PRs for massive architectural overhauls without discussing them in an Issue first.
- Understand that not all PRs will be merged. The maintainer reserves the right to reject contributions that do not align with the project's goals or vision.

Thank you for helping make SafeKey better!
