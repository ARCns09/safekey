# Security Policy

Security is a top priority for SafeKey. We appreciate the efforts of security researchers and the community in helping us maintain the highest standards of safety for our users.

## Supported Versions

We currently provide security updates for the following versions of SafeKey:

| Version | Supported          |
| ------- | ------------------ |
| 3.x.x   | :white_check_mark: |
| 2.x.x   | :white_check_mark: |
| 1.x.x   | :x:                |

Users are always encouraged to update to the latest available release.

## Reporting a Vulnerability

If you discover a security vulnerability in SafeKey, please practice **Responsible Disclosure**. 

**DO NOT** open a public issue on GitHub for security vulnerabilities.

Instead, please report it privately:
1. Contact the maintainer directly via email or through GitHub's private vulnerability reporting feature (if enabled on this repository).
2. Provide a detailed description of the vulnerability, including steps to reproduce it.
3. If possible, provide a proof-of-concept (PoC).

## Expected Response Times

- **Acknowledgment:** We aim to acknowledge receipt of vulnerability reports within 48 hours.
- **Initial Assessment:** We will provide an initial assessment and estimated timeline for a fix within 1 week.
- **Resolution:** Critical vulnerabilities will be addressed as quickly as possible, typically within a few days. Non-critical vulnerabilities will be patched in the next scheduled release.

## Responsible Disclosure Guidelines

When reporting a vulnerability, we ask that you:
- Do not exploit the vulnerability to access user data, disrupt services, or cause harm.
- Give us a reasonable amount of time to patch the issue before making any public disclosure.
- Do not demand financial compensation in exchange for vulnerability reports.

We will credit you in the release notes for discovering and responsibly disclosing the vulnerability if you wish.

Thank you for helping keep SafeKey secure!
